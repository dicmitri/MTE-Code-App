# Code Content Split

## Outcome

On 22 July 2026, the MedTech Code content was moved from one large
`src/data/codeData.json` file into one JSON file per chapter under
`src/data/code/`.

This was a content-neutral migration. No chapter was retyped, reformatted,
summarised, corrected, or generated by AI. A JSON-aware script extracted each
chapter directly from the frozen source text.

The preservation check proves all of the following:

- 23 chapter objects are present in the original order.
- All 67 sections, 43 Q&As, and 335 string values are present.
- Every chapter's raw source slice has its original SHA-256 hash.
- Parsed data is deeply equal to the former monolith.
- Generated section IDs and public routes are unchanged.
- Recombining the chapter slices recreates the exact 232,738-byte monolith with
  SHA-256 `CAA166803AC1EE0CD2B19E0BB1998C6688C53DA8B3FA66F5F3CF75CBA2283EAC`.

Run the archived proof from the app root with:

```powershell
npm run verify:code-migration
```

This command verifies the 2026 migration baseline. It is not an everyday
content check. A later, intentional content edit will correctly make this
historical proof fail. Do not regenerate `src/data/code-manifest.json` merely
to silence that failure.

After that frozen baseline was recorded, all 22 PDF-backed chapters were
reviewed chapter by chapter against the supplied September 2024 PDF. Approved
corrections were then applied to the split files. The current content is
therefore intentionally different from the frozen migration baseline; the
complete correction record is in `pdf-reconciliation.md`.

## Current Files

- `src/data/code/*.json`: the editable chapter files and current Code content.
- `src/data/codeOrder.js`: the explicit chapter order used by the app and tools.
- `src/data/codeData.js`: the compatibility adapter that still exports
  `FULL_CODE_DATA` to the rest of the app.
- `src/data/code-manifest.json`: immutable migration hashes and reconstruction
  metadata.
- `src/data/code-september-2024 (1).pdf`: the supplied September 2024
  authoritative reference PDF.
- `scripts/verify-code-migration.mjs`: the raw and semantic preservation proof.
- `scripts/audit-code-pdf.py`: the read-only PDF comparison tool.
- `code-pdf-audit.json`: the detailed post-split PDF comparison evidence.
- `pdf-reconciliation.md`: the chapter-by-chapter correction policy and review
  log.

## Editing A Chapter

1. Open `src/data/code/`.
2. Choose the file whose name matches the chapter URL. For example,
   `/code/ch4` uses `ch4.json`.
3. Change only the intended text. Do not change `id`, reorder chapters, or run a
   bulk formatter unless that structural change has been separately approved.
4. From the app root, run:

```powershell
npm run validate:data
```

5. Before a release, run:

```powershell
npm run check
```

`npm run validate:data` is the routine check. It verifies chapter files,
chapter IDs, generated section IDs, decision-tree links, icons, Q&As, and quiz
references without requiring content to remain frozen forever.

The CMS opens each chapter as a separate entry. Chapter creation and deletion
are disabled; chapter metadata is hidden; and Code sections and Q&As cannot be
added, removed or reordered. HTML fields use an exact-string editor with an
optional sandboxed preview instead of Markdown conversion. The hosted CMS only
sees files already committed to its GitHub branch. For uncommitted chapter
files, use the local workflow in `docs/cms-guide.md`.

## PDF Audit

The current split files were compared with the supplied 80-page PDF after HTML
styling, PDF page headers, line wrapping, whitespace, and bullet glyphs were
handled conservatively. Punctuation, case, wording, numbers, and apostrophes
remain significant.

The final generated report contains 241 source-backed fields across the 22
PDF-backed chapters. The app-only Version History is excluded:

- 179 match after layout normalization.
- 14 have punctuation differences.
- 7 have case or punctuation differences.
- 41 enter the audit's conservative `substantive_review_required` category.

All 62 non-exact fields were then reviewed by exact audit path using a second
PDF extractor and direct page inspection. The reconciliation log classifies
all 62 as documented PDF typo corrections, extraction or field-boundary
artifacts, app navigation labels or annex-title conventions, semantic HTML
markers, or combinations of those cases. There are no unreviewed substantive
discrepancies.

The authoritative PDF has SHA-256
`9ED658D0C9F858E35576C3905CC4423DF934DB894786F4C12D9A130E0602A148`.
The current corrected split files have aggregate SHA-256
`7CDB58249FAF41A7A7BDBFE6FE7E7089BE496914F2DFDA942AC5B9CEF1C7CB1D`.

Annex III now uses the exact bitmap embedded in PDF page 75 rather than a
redrawn or inferred map. Its source, file hash, dimensions, rendering checks,
and print checks are recorded in `pdf-reconciliation.md`.

To rerun the audit, use a Python environment containing `pypdf`:

```powershell
python scripts/audit-code-pdf.py --output docs/content-migration/code-pdf-audit.json
```

The command only reads the chapter files and PDF. It does not modify either
source.
